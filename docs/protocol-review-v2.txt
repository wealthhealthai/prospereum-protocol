[C1] REWARD RATE MISMATCH -- WHITEPAPER vs DEV SPEC (UNRESOLVED)
-----------------------------------------------------------------
Status: NOT fixed in v2.11. This is the single most important blocker.

The two documents specify incompatible reward rates for partner tiers.

WHITEPAPER Section 6.2 (explicit flat rates per tier):
  Bronze : 8%
  Silver : 10%
  Gold   : 12%

DEV SPEC Section 1.3-1.4 (alphaBase x multiplier):
  alphaBase default = 0.10 (10%)
  Bronze : 0.10 x 1.00 = 10.0%   <-- whitepaper says 8%
  Silver : 0.10 x 1.25 = 12.5%   <-- whitepaper says 10%
  Gold   : 0.10 x 1.60 = 16.0%   <-- whitepaper says 12%

The gap is 2pp for Bronze, 2.5pp for Silver, 4pp for Gold. Gold partners would
receive 33% more rewards under the dev spec than the whitepaper promises.

Additional inconsistency in the DEMAND CAP FORMULA:
  Whitepaper 5.2.2 : E_demand = r_base x NB_total  (uniform 10% of total buys)
  Dev Spec 5.3     : E_demand = SUM(alpha_p x deltaNB_p)  (per-partner alpha)

When partners have different tiers, these produce different caps. A single Gold
partner buying 1,000 PSRE generates:
  Whitepaper: E_demand = 100 PSRE
  Dev Spec  : E_demand = 160 PSRE  (at Gold alpha=16%)

This means the dev spec emits more aggressively than the whitepaper describes.

RESOLUTION OPTIONS:
  Option A: Use whitepaper rates (8/10/12%). Update dev spec:
            alphaBase = 0.08, multipliers = 1.0 / 1.25 / 1.50
            Also update governance bounds to [0.04, 0.12] proportionally.
            Also align demand cap formula: E_demand = alphaBase x NB_total
            (uniform base rate, not per-partner alpha for the cap).

  Option B: Use dev spec rates (10/12.5/16%). Update whitepaper table.
            Accept that demand cap is per-partner-weighted.

  RECOMMENDATION: Option A. The whitepaper is the public-facing document.
  Partners and investors will read it. The rates it promises should be what
  the protocol actually delivers. Conservative rates also reduce emission
  pressure, which is better for long-term scarcity.

ACTION REQUIRED: Jason and Shu decide. I cannot code RewardEngine until this
is resolved -- the choice directly determines what I put in the contract.


[C2] TEAM VESTING CONTRACT -- MISSING FROM SPEC (UNRESOLVED)
------------------------------------------------------------
Status: NOT in v2.11. Still a genesis-critical gap.

Whitepaper Section 3.2 states:
  "Team & Founders: 4,200,000 PSRE -- Minted at genesis to a vesting contract"
  "1-year cliff, 4-year linear vesting, no governance override"

This contract does not appear anywhere in the dev spec contract list (Sections

At genesis, 4.2M PSRE (20% of total supply) must be minted to this contract.
If it does not exist, is deployed incorrectly, or has a bug:

  - Vest duration: 4 years linear from cliff end
  - Beneficiaries: who holds the team allocation (addresses)
  - No governance override: owner cannot change cliff/vest parameters
  - Recommended: use OpenZeppelin VestingWallet or battle-tested equivalent

SECTION 2: SECURITY VULNERABILITIES
================================================================================

[S1] BUY() SANDWICH ATTACK (MEV) -- MEDIUM SEVERITY
----------------------------------------------------
PartnerVault.buy() swaps USDC -> PSRE via Uniswap v3 on Base.
MEV bots on Base monitor the mempool and can sandwich partner buys:
  1. Bot front-runs: buys PSRE before partner
  2. Partner executes buy at inflated price, receives less PSRE
  3. Bot back-runs: sells PSRE for profit

Impact: Partner's cumBuy is lower than it should be -> fewer rewards earned.
Protocol funds are NOT at risk, but partners suffer economic loss.

The dev spec (Section 2.3 Security) says to use ReentrancyGuard but does NOT
mention slippage protection. This is a gap.

MITIGATION: buy() MUST include a minAmountOut parameter passed to the router.
  require(minAmountOut > 0, "slippage protection required");
  params.amountOutMinimum = minAmountOut;

Partners should also be advised to use tight slippage tolerances (0.5-1%).
Consider integrating with Flashbots Protect RPC on Base for large buys.


[S2] EPOCH LIVENESS -- HIGH SEVERITY (OPERATIONAL)
--------------------------------------------------
finalizeEpoch() is callable by anyone, one epoch at a time, strictly sequential.
If no one calls it for epoch N, epoch N+1 can never be finalized. All rewards
are blocked permanently until epoch N is finalized.

This is NOT a smart contract bug -- it is an operational dependency.
The protocol has no built-in keeper. Without external infrastructure,
rewards simply do not flow.

Attack surface: A malicious actor cannot permanently prevent finalization (anyone
can call it), but they can cause delays. More realistically, operational neglect
(no keeper deployed) causes the same problem.

MITIGATION:
  1. Deploy Chainlink Automation or a custom keeper bot at launch (Day 1)
  2. Add a catchUpEpochs(uint256 maxEpochs) function for gap recovery:
       for i in range(lastFinalizedEpoch+1, min(currentEpoch, lastFinalizedEpoch+maxEpochs+1)):
           _finalizeEpochInternal(i)
  3. Document the keeper SLA -- what is acceptable lag?
  4. Consider incentivizing callers: small PSRE reward for calling finalizeEpoch()

Note: The spec says "callable by anyone after epoch ends" which is correct and
avoids centralization. The risk is purely operational.


[S3] REWARDENGINE SINGLE POINT OF FAILURE -- HIGH SEVERITY
----------------------------------------------------------
RewardEngine is the SOLE authorized minter of PSRE (Section 2.1).
If RewardEngine is compromised -- via a buggy UUPS upgrade, or a multisig
key compromise -- an attacker can mint up to the full 12.6M emission reserve
to any address in a single transaction.

The spec says UUPS + multisig + timelock for early-phase RewardEngine.
This is the right direction but needs explicit parameters:

  - Multisig: Gnosis Safe, minimum 3-of-5, hardware wallets only
  - Timelock: minimum 48 hours on mainnet (72 hours preferred)
  - Mint rate limiter: PSRE.mint() should revert if called more than once
    per epoch window, OR if single call mints more than E0_MAX (25,200 PSRE)
    This limits blast radius even if RewardEngine is compromised

RECOMMENDATION: Add a mint rate limiter directly to the PSRE token contract:
  mapping(uint256 => uint256) public epochMinted;  // epochId -> amount minted
  function mint(address to, uint256 amount) external onlyRewardEngine {
      uint256 epochId = (block.timestamp - genesisTimestamp) / 7 days;
      require(epochMinted[epochId] + amount <= E0_MAX, "epoch mint cap exceeded");
      epochMinted[epochId] += amount;
      _mint(to, amount);
  }
This adds a second layer of defense independent of RewardEngine logic.


[S4] PARTNER WALLET SQUATTING -- MEDIUM SEVERITY (NEW IN V2.11)
--------------------------------------------------------------
Dev Spec v2.11 changed partner identity from partnerId (uint) to partnerAddress
(wallet address). PartnerVaultFactory now maps: partnerAddress -> vaultAddress.

This introduces a new attack surface:
  - Any wallet can register as a partner and create a vault
  - An attacker pre-registers 1000 wallets cheaply on Base (gas is ~$0.001/tx)
  - Each registered vault participates in the RewardEngine finalization loop
  - Gas cost for finalizeEpoch() grows O(n) with partner count
  - At 1000 fake partners, finalization may exceed block gas limit

Impact:
  - Griefing: attacker pays ~$1 to make finalization permanently impossible
  - Reputation: protocol appears broken to legitimate partners

MITIGATION:
  1. Registration fee: require payment (e.g., 0.005-0.01 ETH) to create vault
     This makes 1000 wallet attack cost $5-10, still cheap
  2. Better: admin/DAO whitelist for v1 launch. Only approved partners can
     register. Transition to permissionless in v2 with higher registration fee.
  3. Hard cap: MAX_PARTNERS constant (e.g., 200). DAO can increase it.
  4. Gas guard in finalizeEpoch():
       require(registeredVaults.length <= MAX_PARTNERS, "too many partners");


[S5] EPOCH BOUNDARY BUY MANIPULATION -- LOW-MEDIUM SEVERITY
-----------------------------------------------------------
A partner can monitor block timestamps and place a large buy() in the final
block before epoch end, maximizing their deltaNB for that epoch. After
finalizeEpoch(), they stop buying.

Existing protection: EMA with theta=1/13 means a spike only contributes 7.69%
to their rolling score. Tier assignment is resistant to single-epoch spikes.

Remaining risk: The spike still earns rewards proportional to w_p = alpha_p x
deltaNB for that epoch. Even at Bronze (alpha=10%), a $10,000 spike buy earns
~$1,000 in rewards that epoch. The attacker is providing real capital (they
bought PSRE), so this may be acceptable by design.

RECOMMENDATION: Accept for v1. Document explicitly that large one-time buys
earn large one-time rewards (this is by design -- real demand = real rewards).
Consider a rolling-average cap in v2 if gaming becomes prevalent.

SECTION 3: DESIGN CONCERNS
================================================================================

[D1] PARTNER WALLET MIGRATION NOT POSSIBLE (NEW IN V2.11)
---------------------------------------------------------
With partnerAddress as identity, a partner who needs to rotate their wallet
(key compromise, operational change, company restructuring) cannot migrate
their vault or EMA history to a new address. Their options are:
  - Keep the compromised wallet (security risk)
  - Register a new address and start EMA from zero (loss of tier status)

RECOMMENDATION: Add an optional transferOwnership(address newOwner) to
PartnerVault that allows the current partnerOwner to transfer control.
The vault address and EMA history in RewardEngine stay the same -- only
the owner (who can call buy/distribute) changes. This does NOT require
a PartnerRegistry redesign.


[D2] LP STAKING: 1 LP TOKEN != 1 PSRE IN ECONOMIC VALUE
-------------------------------------------------------
Dev Spec v2.11 Section 2.4 and 7 state: "LP and PSRE staking treated equally,
stakeTime = stakeAmount x stakingDuration, no weighting multiplier."

This is an intentional design choice to incentivize LP provision without an
oracle. The tradeoff is accepted. However, it should be documented:

A Uniswap v3 LP token contains BOTH PSRE and USDC, making it worth more than
1 PSRE in capital terms. As PSRE price rises, LP tokens become increasingly
more valuable relative to PSRE. Over time, LP stakers will dominate the
staking pool and crowd out pure PSRE stakers.

This is arguably desirable (more liquidity = better protocol health), but
pure PSRE holders may feel the staking incentive is unfair. This should be
disclosed in documentation.

RECOMMENDATION: Note in the whitepaper that LP staking intentionally gives
capital-efficient participants better staking yield per PSRE-equivalent
deployed, as an incentive for liquidity provision.


[D3] NO MINIMUM STAKE DURATION DEFINED
--------------------------------------
Whitepaper Section 6.4: "Users may lock PSRE tokens for a minimum duration"
Dev Spec: No minimum lock period is defined anywhere.

Without a minimum, a user can stake 1 block before epoch end and earn a tiny
but non-zero reward. Time-weighting makes this economically negligible, but
the language "minimum duration" in the whitepaper promises something that
does not exist in the spec.

RESOLUTION: Either
  A) Define a minimum: e.g., 24 hours, or 1 epoch minimum stake period, OR
  B) Remove "minimum duration" from the whitepaper and rely solely on
     time-weighting as the anti-flash-stake mechanism (simpler, less gas)


[D4] GENESIS LIQUIDITY STRATEGY NOT SPECIFIED
--------------------------------------------
Whitepaper Section 4: "Liquidity seeded solely from treasury allocation."
  Bootstrap Liquidity: 1,050,000 PSRE (5% of supply)
  This is the only reference to how liquidity is bootstrapped.

Missing:
  - What price to seed the PSRE/USDC Uniswap v3 pool
  - What tick range to use (required for concentrated liquidity)
  - How much of the 1,050,000 PSRE to deploy vs hold in reserve
  - Whether to use full-range or concentrated liquidity position

Risk: Seeding a pool at the wrong price allows instant arbitrage loss.
Seeding with too little depth causes first partner buy to move price dramatically.

RECOMMENDATION: Define the deployment plan:
  - Launch price (e.g., $0.10 PSRE/USDC)
  - Initial pool depth in USD terms
  - Tick range (full-range for simplicity at launch, concentrate later)
  - Add this to the deployment checklist alongside the contract deployments


[D5] TREASURY ALLOCATION BOUNDARIES ARE NOT ENFORCED ON-CHAIN
-------------------------------------------------------------
Whitepaper Section 3.2 defines three treasury sub-allocations:
  Ecosystem Growth : 1,680,000 PSRE (8%)
  DAO Treasury     : 1,470,000 PSRE (7%)
  Bootstrap Liq    : 1,050,000 PSRE (5%)

All three are minted to a single "Treasury Wallet (SAFE)". There is no on-chain
mechanism preventing the multisig from using Bootstrap Liquidity tokens for
ecosystem grants, or DAO Treasury tokens for liquidity.

This is a governance/trust concern, not a technical exploit. Partners and
investors who read the whitepaper will expect the allocations to be honored.

RECOMMENDATION: Disclose clearly in documentation that sub-allocations are
social/governance commitments, not on-chain enforced. For v2: consider
separate Gnosis Safe wallets per allocation category to add accountability.


[D6] SUMR PRECISION DRIFT OVER LONG TIME HORIZON
------------------------------------------------
The EMA formula:
  R_new = (R_old * (1e18 - theta) + deltaNB * theta) / 1e18

Each division by 1e18 truncates (rounds down). Over 1000+ epochs, the
truncation accumulates. sumR is updated incrementally:
  sumR = sumR - R_old + R_new

If R_new is consistently rounded down, sumR drifts slightly below the true
sum. Share calculations (s = R_new * 1e18 / sumR) become slightly inflated,
which could push partners across tier thresholds incorrectly.

Impact: Very small per epoch (sub-wei). Material only after 10+ years.
Low priority for v1.

RECOMMENDATION: Add recalculateSumR() function that recomputes sumR from
scratch by iterating all registered vaults. Can be called by governance
or anyone as a maintenance function. Emit a SumRRecalculated event.


[D7] NO PAUSE MECHANISM SPECIFIED
---------------------------------
Neither document mentions an emergency pause capability.

If a critical bug is found in RewardEngine after mainnet launch, there is no
way to stop new minting while a fix is prepared. The only option would be to
have the multisig immediately replace RewardEngine (if UUPS), but this requires
the timelock delay to pass (48-72 hours), during which the exploit continues.

RECOMMENDATION: Add Pausable to RewardEngine.
  - Pause halts: finalizeEpoch() only (stops new minting)
  - Pause does NOT halt: claimPartner(), claimStake() (users keep access to
    already-earned rewards)
  - Pause authority: Gnosis Safe with NO timelock (emergencies require speed)
  - Unpause authority: Same Safe, or require DAO vote for longer pauses

  ---------    -----------    -------    ---------------

  1,260,000    0.100          0.810       10,206
  3,150,000    0.250          0.563        7,088
  6,300,000    0.500          0.250        3,150
  9,450,000    0.750          0.063          788
  11,340,000   0.900          0.010          126
  12,474,000   0.990          0.000            1
  12,600,000   1.000          0.000            0

OBSERVATION: After 90% depletion, weekly max emission is only 126 PSRE.
At the upper E0 bound (25,200/week), these numbers double but the shape
is the same. The tail is asymptotically approaching zero -- the protocol
can theoretically never fully exhaust the reserve.

EMA ANTI-MANIPULATION ANALYSIS (theta = 1/13)

  Epochs ago    Influence on current R
  ----------    ----------------------
  0 (current)   7.69%
  1             7.10%
  4             5.58%
  13            3.54%
  26            1.57%
  52            0.37%

OBSERVATION: A single-epoch spike influences current R by at most 7.69%.
After 6 months (26 epochs), its influence drops below 2%. The EMA is
appropriately resistant to single-epoch gaming for tier assignment.

NOTE: The EMA protects TIER, not raw deltaNB rewards. A large spike still
earns large rewards in the epoch it occurs (by design -- real demand is real).
